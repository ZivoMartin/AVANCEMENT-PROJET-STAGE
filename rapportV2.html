The recorder/replayer software

A recorder/replayer is a software that first records a sequence of actions performed when interacting with the browser, then converts this sequence of actions into a script.
Once executed, the script will call the replayer, which will read the script and execute the actions one by one on the browser. This testing software allows one to quickly and automatically test a web application rather than having to test all the application's functions each time it is modified. All you have to do is run the script, which is very convenient.

However, recorders do have their weaknesses. The recorder listens to JavaScript events and records them.
However, some events are not detected by the recorder, which leads to problems during replay. The aim of this report is therefore to compare four test programs: Cypress, Selenium, Playwright and Chrome, and to determine the strengths and weaknesses of each.

In this study, we first selected JavaScript events divided into two categories: those in the "mouse events" class and those in the "keyboard events" class. This table compares the management of keyboard and mouse events for each software. It does not deal with sequences of events (scenarios), but rather with single events.

Now that we know which events are taken into account by each recorder, it's easy to determine whether or not a given scenario is recordable: if we take X to be a scenario and (x1, x2, ... , xn) to be the set of events included in X, X is recordable if x1, x2, ..., xn are all events taken into account by the recorder. So let's look at the second part of these tools, replay.
If we assume that the recording only contains only actions taken into account by the software, there are only two possible causes of failure. The first is likely to to occur in every program, and concerns CSS selectors. A bad selector will result in a bad replay.
The second cause of error is likely to occur only on Cypress and Playwright, and involves incorrect syntax in the script. This problem does not arise in Selenium/Chrome, as these two programs have an instant replay option which does not involve scripting.
However, it is possible to export a sequence of interactions to a script in JavaScript, Python, Julia, etc. However, the ability to export the recording is not taken into account in this study. We therefore need to test two things: firstly, the CSS selection algorithm, and secondly, the quality of the syntax of the Cypress and Playwright scripts.

Let's start with CSS selectors. Testing the ability of a given software to obtain the CSS selector is simple. You go to a website, click on an element, then if you're on Selenium/Chrome you can play it back instantly, otherwise you export the script. If the syntax of the script is wrong, we correct it. To be as precise as possible, we're going to make 50 clicks on 10 different sites and count the number of accepted clicks. The 10 sites in question will be : Google, Amazon, Openai, Microsoft, Globo, Wikipedia, Ebay, Dzen, Office, Samsung. For the selection of the different sites, we simply did not take the sites that require a connection like instagram, facebook... Because in general, the replayer is not accepted by the site.
